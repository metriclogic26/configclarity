nginx Config Validator

Paste nginx.conf — validate routing, SSL redirects, and proxy_pass issues

What this validator checks

  • Missing SSL redirect — server block on port 80 without return 301 https://
  • proxy_pass to localhost — won't work inside Docker containers
  • Duplicate server_name — same hostname in multiple server blocks
  • Conflicting ALLOW/DENY rules — both rules matching same source
  • Redundant rules — specific IP rule made irrelevant by Anywhere rule
  • URL matching — test a URL and see which location block matches

Common nginx mistakes

1. Missing HTTP to HTTPS redirect

❌ No redirect — HTTP visitors never get HTTPS 
server {
    listen 80;
    server_name example.com;
    location / {
        proxy_pass http://app:3000;  # Should redirect to HTTPS
    }
}
✅ Redirects HTTP to HTTPS 
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name example.com;
    location / {
        proxy_pass http://app:3000;
    }
}

2. proxy_pass to localhost in Docker

❌ localhost resolves to the nginx container itself 
location / {
    proxy_pass http://localhost:3000;  # Wrong in Docker
}
✅ Use Docker service name 
location / {
    proxy_pass http://app:3000;  # "app" = compose service name
}

Frequently Asked Questions

Can I test which location block matches a URL?

Yes. Enter a test URL in the optional URL field and the Reverse Proxy Mapper highlights which location block or Traefik router would match it, along with the proxy_pass destination.

Does it validate nginx syntax errors?

The browser-based validator checks structural patterns (missing SSL redirect, localhost proxy_pass, duplicate server names) but not full nginx syntax. Run nginx -t on your server for full syntax validation. The Reverse Proxy Mapper focuses on routing logic and security issues.

Does it support Traefik labels?

Yes. Paste docker-compose.yml with Traefik labels and the validator auto-detects Traefik format. It supports both Traefik v2 and v3, with v3-specific checks for deprecated middleware syntax.
