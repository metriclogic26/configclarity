What makes cron expressions tricky

Day of week: 0 and 7 are both Sunday

0 0 * * 0   # Sunday (0)
0 0 * * 7   # Also Sunday (7) — both valid

Step values

*/15 * * * *    # Every 15 minutes: :00, :15, :30, :45
0 */6 * * *    # Every 6 hours: 00:00, 06:00, 12:00, 18:00
0 8-18/2 * * * # Every 2 hours between 8am-6pm

Lists and ranges

0 9,17 * * *     # At 9:00 AM and 5:00 PM
0 9 * * 1,3,5   # Monday, Wednesday, Friday at 9:00 AM
0 0 1-7 * *     # First 7 days of every month

Frequently Asked Questions

Does cron use the server's timezone?

Yes. Cron runs in the system timezone (usually UTC on servers). The ConfigClarity Cron Visualiser lets you set UTC, local, or a custom timezone offset so you can verify schedules in the context of your server's clock.

Why is my cron job not running even though the expression looks correct?

Common causes: the cron daemon isn't running (systemctl status cron), the script has a PATH issue (cron runs with minimal PATH), the script isn't executable (chmod +x script.sh), or output is going nowhere. Add >> /tmp/cronlog.txt 2>&1 to capture all output.

What does @reboot do?

@reboot command runs once when the cron daemon starts (typically at system boot). It's equivalent to running the command once on startup. There's no guarantee of timing relative to other services — use systemd units for ordering guarantees.
